**Past-paper TODO follow-up (same session, 2026-06-25):** User opened `PastPaper_0478_w25_23.md` in the IDE and asked what to do about the remaining `% TODO: ...no matching MS entry found` lines. Found and fixed two more real bugs rather than leaving them:
1. `parse_qp()` only matched a sub-part/sub-sub-part marker at the very start of a line -- when CIE prints a marker on the *same* physical line as its parent (e.g. `"4 (a) Tickets for a theme park..."` or `"(a) (i) Identify..."`), the child marker was silently swallowed into the parent's question text, so no QP label ever existed for it and its real MS answer sat orphaned in "Unmatched Mark Scheme Entries". 2. `is_stem_only` assumed a scenario-intro stem could never itself end in `)`, so a lettered stem with roman-numeral children (`4(c)` before `4(c)(i)/(ii)`, `9(b)` before `9(b)(i)/(ii)`) wrongly showed a TODO instead of the correct "(Scenario introduction...)" note. Paused and asked the user whether to fix now or document as a known limitation; user chose to fix it now.
- The real root cause of the orphaned-label noise (`"0: 1"`, `"7686: "`, `"300: abnormal"`, etc.) turned out to be nested sub-tables (truth-table grids, tick-grids) that `find_tables()` returns as their own separate table objects with an unrelated column count that's still coincidentally a multiple of 3 -- a bare digit in their own data (e.g. a truth table's "0"/"1" output column) could coincidentally collide with a real question number. Fixed by requiring a table to contain its own literal "Question" header cell to count as the genuine grid at all. This single change both makes cross-page accumulation safe (a same-label match across two genuine top-level tables is now guaranteed to be a real continuation, since question numbers are unique within one paper) and eliminates the nested-table noise everywhere, not just where it was found.
